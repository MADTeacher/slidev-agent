# HTML-презентации, экспорт и проверка

HTML — рабочий исходник этого навыка. Статические экспорты строятся из HTML, кроме редактируемого PPTX: для него с самого начала нужен HTML с ограничениями ради корректного экспорта.

## Первое решение: формат сдачи

Спроси или выведи из задачи:

- только HTML: максимальная свобода;
- редактируемый PPTX: исходник с ограничениями, см. `pptx-authoring.md`; он должен быть безопасен для Windows/macOS PowerPoint, а не только для локального браузера;
- PDF: печать/экспорт из браузера, высокая визуальная точность;
- PNG: скриншоты по слайдам;
- SVG: нативный SVG или путь SVG-ассета; надёжен только для простых нативных векторных слайдов или специально подготовленных SVG-материалов, а не для произвольного HTML.

## Рабочая директория и Node

Для каждого проекта презентации создавай отдельную директорию, например:

```text
my-deck/
├── package.json
├── asset-manifest.json
├── node_modules/
├── slides/
├── exports/
└── scripts/
```

Все Node/npm-окружение, то есть среда JavaScript и ее пакетный менеджер, а также временные скрипты агента должны жить внутри этой директории. Не устанавливай и не линкуй `node_modules` в корень репозитория, в директорию навыка или в общую рабочую папку вне конкретной презентации.

Сразу скопируй весь каталог `scripts/` навыка в `scripts/` директории презентации. Не копируй отдельные файлы: у экспортов есть соседние зависимости, например `export_deck_pptx.mjs` использует `html2pptx.js`.

В рабочей директории презентации обязан быть `asset-manifest.json`. Для дека без предметных визуалов можно указать пустой массив `visuals`, но файл все равно нужен: штатные PNG/PDF/PPTX-экспорты запускают `asset_gate_check.mjs` перед созданием артефактов.

Скрипты навыка запускай из директории презентации:

```bash
node scripts/export_deck_png.mjs --slides slides --out exports/png
```

Не вызывай экспортные скрипты по `../scripts/...` или абсолютным путям к навыку. Презентационная папка должна быть самодостаточной: локальный Node, локальные `node_modules`, локальные HTML-слайды, локальные копии скриптов навыка, локальные экспорты.

## Размеры canvas

Canvas здесь означает рабочую область слайда. По умолчанию:

- HTML/PDF/PNG: `1920x1080px`;
- редактируемый PPTX: `960x540pt`, соответствует широкому формату PowerPoint.

Другие размеры используй только по требованию площадки, шаблона или задачи.

## 2-слайдовое направление до массовой сборки

Перед созданием всей презентации сделай:

1. обложку;
2. репрезентативный плотный контентный слайд.

Эти два слайда фиксируют визуальную грамматику: типографику, интервалы, палитру, изображения, графики, footer, то есть нижнюю служебную строку, и экспортные ограничения.

## Многофайловая архитектура

Многофайловая архитектура означает, что презентация разделена на несколько файлов. Используй её по умолчанию для серьезных презентаций.

```text
deck/
├── index.html
├── shared/
│   ├── tokens.css
│   └── components.css
└── slides/
    ├── 01-cover.html
    ├── 02-agenda.html
    └── 03-content.html
```

Стартуй с `assets/deck_index.html` и отредактируй `DECK_MANIFEST`.

Плюсы:

- ошибка одного слайда не ломает всю презентацию;
- каждый слайд можно открыть отдельно;
- CSS и layout-проблемы, то есть проблемы компоновки, проще изолировать;
- разные люди могут работать над разными слайдами.

## Однофайловая архитектура

Однофайловая архитектура означает, что вся презентация лежит в одном файле. Для маленьких презентаций и быстрых черновиков используй `assets/deck_stage.js`.

```html
<script src="assets/deck_stage.js"></script>
<deck-stage>
  <section class="slide">...</section>
  <section class="slide">...</section>
</deck-stage>
```

Каждый `section` должен иметь фиксированные размеры и `overflow: hidden`.

## Грамматика слайда

Базовый слайд:

```html
<section class="slide">
  <p class="kicker">Раздел</p>
  <h1>Заголовок слайда</h1>
  <p class="lead">Главная мысль.</p>
  <div class="content-grid">...</div>
  <p class="slide-number">03</p>
</section>
```

Правила:

- одна главная мысль на слайд;
- основной текст обычно `24px` или больше на `1920x1080`;
- подписи обычно `18px` или больше;
- поля и номера слайдов стабильны;
- роли слайдов чередуются.

## React/Babel

Используй React только когда он уменьшает повторение: компоненты графиков, повторяемые макеты слайдов, canvas для вариантов или общие разделы, которые строятся из данных. Для маленьких презентаций часто достаточно обычного HTML/CSS без фреймворка.

Подключение:

```html
<script src="https://unpkg.com/react@18.3.1/umd/react.development.js" crossorigin="anonymous"></script>
<script src="https://unpkg.com/react-dom@18.3.1/umd/react-dom.development.js" crossorigin="anonymous"></script>
<script src="https://unpkg.com/@babel/standalone@7.29.0/babel.min.js" crossorigin="anonymous"></script>
```

Компоненты подключай как Babel-скрипты, которые Babel переведет в обычный JavaScript:

```html
<script type="text/babel" src="components.jsx"></script>
<script type="text/babel" src="deck.jsx"></script>
```

Каждый Babel-файл компилируется отдельно. Общие компоненты экспортируй в `window`, чтобы они были доступны другим файлам.

```jsx
function MetricCard({ label, value }) {
  return (
    <div className="metric-card">
      <p>{label}</p>
      <strong>{value}</strong>
    </div>
  );
}

window.MetricCard = MetricCard;
```

Держи данные слайдов в объектах, совместимых с JSON:

```js
window.DECK_DATA = {
  metrics: [
    { label: 'Revenue', value: '$12.4M' },
  ],
};
```

Используй уникальные имена компонентов и стилей для каждой презентации. Избегай глобальных `styles`, `data` или `Card`, если загружается несколько скриптов.

Редактируемый PPTX читает DOM после отрисовки в браузере. Если нужен PPTX, финальный текст должен быть в семантических текстовых тегах, а эффекты, которые работают только в браузере, лучше исключить.

Если нужен редактируемый PPTX, браузерная PNG-проверка обязательна, но не достаточна. До экспорта должны быть безопасные основные шрифты, UTF-8 meta на каждом слайде и читаемые эмоджи, если они используются как учебные подсказки.

## Заметки докладчика

Используй `aside.notes` или JSON-блок заметок. Заметки докладчика помогают выступать, а не дублируют видимый текст.

```html
<aside class="notes">
  Перед следующим слайдом объяснить, почему изменилась метрика.
</aside>
```

## PDF-экспорт

Многофайловая презентация:

```bash
node scripts/export_deck_pdf.mjs --slides slides --out exports/deck.pdf
```

Однофайловая презентация `deck-stage`:

```bash
node scripts/export_deck_stage_pdf.mjs --html deck.html --out exports/deck.pdf
```

PDF должен сохранять визуальную точность и текст, который можно выделить и найти поиском, если браузер может напечатать текст как вектор.

## PNG-экспорт

PNG подходит для проверки, миниатюр и статичной отправки слайдов.

Многофайловая презентация:

```bash
node scripts/export_deck_png.mjs --slides slides --out exports/png
```

Однофайловая презентация `deck_stage`:

```bash
node scripts/export_deck_stage_png.mjs --html deck.html --out exports/png --slides 12
```

Проверь загрузку шрифтов до скриншотов. Скрипт ждет момента, когда браузер закончил активную работу, и `document.fonts.ready`.

PNG-экспорт всех слайдов — обязательная проверка качества перед финальной сдачей, даже если пользователь попросил только PPTX или PDF. После экспорта:

1. Проверь, что число PNG равно числу HTML-слайдов или `<section>` в `deck_stage`.
2. Проверь, что `asset-manifest.json` прошел `node scripts/asset_gate_check.mjs --slides slides --manifest asset-manifest.json --mode preexport`.
3. Выполни последовательный проход по `slide-qa.md` и веди `exports/slide-qa.md`.
4. Исправь все обрезания, наложения, пустые картинки, съехавшие подписи, плохую читаемость и расхождения между слайдами.
5. Повтори экспорт PNG и проход `1..N` до чистого результата после последней правки.
6. В финальном ответе назови путь к PNG-папке, путь к `exports/slide-qa.md` и статус проверки всех слайдов.

## Нативный SVG и путь SVG-ассета

SVG не является универсальным экспортом произвольного HTML. Используй только один из проверяемых путей:

- сделай слайд или ключевое смысловое изображение как нативный SVG, если нужен настоящий вектор;
- экспортируй графики/диаграммы как SVG-материалы и вставляй их в HTML-презентацию;
- для обложек, орнаментов, иконок, схем и абстрактных векторных сцен используй `presentation-svg.md`, routed SVG-референсы, шаблоны из `assets/templates/` и проверку через `scripts/presentation_svg_pipeline.mjs`;
- для произвольных HTML-слайдов сдавай PDF/PNG и честно объясняй, почему SVG будет с потерями или нередактируемым.

## Редактируемый PPTX

Команда:

```bash
node scripts/export_deck_pptx.mjs --slides slides --out exports/deck.pptx
```

Сначала прочитай `pptx-authoring.md`. Если исходная презентация была сделана без ограничений для PPTX, сначала перестрой HTML в версию с такими ограничениями.

PPTX не отменяет проверку качества через PNG. Сначала добейся чистого HTML/PNG-рендера всех слайдов, затем экспортируй PPTX штатным скриптом навыка и проверь редактируемость текста.

Если визуальный язык держится на небезопасных шрифтах или сложных браузерных эффектах, не обещай такой редактируемый PPTX как кроссплатформенный. Правильный путь: PDF/PNG или ограниченная PPTX-версия.

## Проверка результата

Каждую HTML-презентацию нужно проверить визуально и технически до сдачи.

Чеклист:

1. HTML открывается в браузере.
2. В консоли нет релевантных ошибок.
3. Шрифты и изображения загружаются.
4. Ключевые слайды читаются в целевом размере.
5. Все слайды экспортированы в PNG через `node scripts/export_deck_png.mjs` для multi-file deck или `node scripts/export_deck_stage_png.mjs` для однофайлового `deck_stage`.
6. `exports/slide-qa.md` содержит строку на каждый PNG, а последний проход `1..N` имеет статус `чистый`.
7. Запрошенные экспорты созданы локальными копиями штатных скриптов навыка.
8. Если был запрошен PPTX, файл создан через `node scripts/export_deck_pptx.mjs` и текст проверен на редактируемость.

Базовая проверка браузера:

```bash
node scripts/verify.mjs deck/index.html
```

Для `deck-stage` файла:

```bash
node scripts/verify.mjs deck.html --slides 12
```

Проверка viewport, то есть области просмотра браузера:

```bash
node scripts/verify.mjs deck/index.html --viewports 1920x1080,1440x900
```

Обязательная проверка качества всех слайдов через PNG для multi-file deck:

```bash
node scripts/export_deck_png.mjs --slides slides --out exports/png
```

Для однофайлового `deck_stage`:

```bash
node scripts/export_deck_stage_png.mjs --html deck.html --out exports/png --slides 12
```

Для визуального прохода используй `slide-qa.md`. В первую очередь ищи:

- обрезанный текст;
- плохое выравнивание;
- нечитаемые подписи графиков;
- пропавшие изображения;
- съехавшие footer, то есть нижние служебные строки, или номера;
- несовместимые отступы между слайдами.

## Проверка экспортов

Используй только локальные копии скриптов навыка:

- `node scripts/export_deck_png.mjs` для PNG;
- `node scripts/export_deck_stage_png.mjs` для PNG из однофайловой архитектуры;
- `node scripts/export_deck_pdf.mjs` для PDF из многофайловой архитектуры;
- `node scripts/export_deck_stage_pdf.mjs` для PDF из однофайловой архитектуры;
- `node scripts/export_deck_pptx.mjs` для редактируемого PPTX;
- `scripts/html2pptx.js` только как низкоуровневую часть PPTX-пути навыка.

Не используй PowerPoint, Keynote, LibreOffice, Quick Look, `open`, `osascript`, автоматизацию рабочего стола или альтернативные самописные экспортеры для финальных экспортов.

PDF:

- число страниц совпадает с числом слайдов;
- фоны видны;
- текст можно выделить и найти поиском, когда это ожидается;
- нет пустой страницы в конце.

PNG:

- число файлов совпадает с числом слайдов;
- изображения достаточно четкие для проверки;
- прозрачные ассеты отображаются корректно;
- `exports/slide-qa.md` показывает чистый проход `N/N` после последнего экспорта.

Нативный SVG и путь SVG-ассета:

- обещай SVG-качество только для слайдов, которые изначально сделаны как ограниченный вектор;
- для сложной художественной SVG-сцены сначала прочитай `presentation-svg.md`;
- запусти `node scripts/presentation_svg_pipeline.mjs validate <file.svg>`;
- для визуальной сдачи запусти `node scripts/presentation_svg_pipeline.mjs audit <file.svg> <preview.png> --width 1920 --height 1080`;
- открой PNG-превью и проверь читаемость как слайдовый материал;
- назови растеризованные или нередактируемые части, если они есть.

PPTX:

- текст редактируется;
- изображения не пропали;
- границы слайда соблюдены;
- визуальные упрощения намеренные.

## Частые сбои

Пустой или частичный рендер:

- неверный локальный путь;
- не хватает зависимости;
- ошибка синтаксиса JSX;
- компонент не прикреплен к `window` при отдельных Babel-файлах.

Неверный шрифт:

- проверь пути `@font-face`;
- жди `document.fonts.ready`;
- не полагайся на шрифт, которого нет в среде экспорта.

Съехал layout, то есть компоновка:

- проверь `box-sizing: border-box`;
- проверь ширину и высоту слайда;
- посмотри grid tracks, то есть дорожки сетки, и absolute-positioned элементы, то есть элементы с абсолютным позиционированием;
- проверьте, использует ли экспорт `px` или `pt`.

## Финальная сдача

Сообщай только важное:

- какие файлы созданы;
- какие проверки запущены;
- где лежат PNG всех слайдов, где лежит `exports/slide-qa.md` и сколько слайдов прошло чисто;
- важные оговорки;
- на каких слайдах еще нужны данные пользователя.
