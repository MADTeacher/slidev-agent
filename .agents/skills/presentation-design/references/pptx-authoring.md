# Создание редактируемого PPTX

Редактируемый PPTX - это перевод HTML-объектов в объекты PowerPoint, а не скриншот. Поэтому исходный HTML с самого начала должен быть ограниченным (`constrained`): с понятными размерами, позициями и простыми правилами верстки.

Редактируемый PPTX здесь означает кроссплатформенный PowerPoint-совместимый результат для Windows/macOS, а не просто HTML, который хорошо выглядит в локальном браузере.

## Размер

Используй:

```css
body {
  width: 960pt;
  height: 540pt;
  margin: 0;
}

.slide {
  width: 960pt;
  height: 540pt;
  position: relative;
  overflow: hidden;
}
```

Это соответствует широкому формату PowerPoint.

## Жесткие ограничения

### 0. Кроссплатформенная текстовая безопасность обязательна

Для редактируемого PPTX используй только такие основные текстовые шрифты:

- `Arial`
- `Verdana`
- `Trebuchet MS`
- `Georgia`
- `Times New Roman`
- `Courier New`

Рекомендуемые роли:

- sans: `Arial`
- serif: `Georgia` или `Times New Roman`
- mono: `Courier New`

Не используй как основной текстовый шрифт:

- `Avenir`
- `Avenir Next`
- `SF Pro`
- `San Francisco`
- `Helvetica Neue`
- `Inter`
- `-apple-system`
- `system-ui`
- `BlinkMacSystemFont`
- `Segoe UI`
- `Calibri`
- `Aptos`
- локальный или брендовый шрифт без честного принятия непаритета или без перевода такого слоя в растр

Каждый PPTX-safe HTML-слайд обязан объявлять:

```html
<meta charset="utf-8" />
```

### 1. Текст - только в семантических текстовых тегах

Текст должен быть в `p`, заголовочных тегах, `ul` или `ol`, а не напрямую внутри `div`.

```html
<!-- Плохо -->
<div>Общая выручка</div>

<!-- Хорошо -->
<div class="label"><p>Общая выручка</p></div>
```

### 2. Визуальные контейнеры отдельно от текста

Фон (`background`), рамка (`border`) и тень (`shadow`) должны быть на контейнере `div`. Текстовые теги держат только текст.

### 3. Избегай CSS без аналога в PPTX

Не используй градиенты (`gradients`), фильтры (`filters`), сложные маски (`complex masks`) и неподдерживаемые режимы смешивания (`unsupported blend modes`) в редактируемых слайдах. Если эффект обязателен, отрисуй его как слой с картинкой (`image layer`), а текст рядом оставь редактируемым (`editable`).

### 4. Изображения через `img`

Не полагайся на `background-image` для объектов, которые должны появиться в PPTX.

```html
<img class="hero" src="hero.png" alt="" />
```

## Растровые изображения в редактируемом PPTX

PNG/JPG/WebP допустимы и ожидаемы для содержательных иллюстраций в редактируемом PPTX. Редактируемость PowerPoint относится к тексту, фигурам, таблицам, графикам и слоям макета; она не требует заменять людей, одежду, предметы, сцены или учебные изображения на SVG.

Редактируемый PPTX не дает права обходить `asset-gate.md` и `content-assets.md`. Редактируемость текста не требует векторной подмены предметных изображений, PNG с SVG-происхождением, HTML/CSS-рисунков, canvas-рендера или скриншота векторной имитации.

Если слайду нужен узнаваемый объект, сначала используй растровый путь из `asset-gate.md` и `content-assets.md`: локальный файл, материалы пользователя, открытый источник или генерация через `imagegen`. Вставляй итоговый файл через `<img>`, а подписи, вопросы, словарь, стрелки и выноски держи отдельными редактируемыми слоями. Если реального PNG/JPG/WebP нет, остановись и запроси материал, поиск, генерацию или явное одобрение запасного варианта.

### 5. Верстка должна быть измеримой

Используй явные размеры, позиции и простые структуры grid/flex. Проверяй, что контент не выходит за границы `body`.

### 6. Эмоджи проверяются как обычный видимый текстовый слой

Эмоджи можно оставлять в редактируемом тексте, если они помогают роли слайда и не ломают верстку. Проверяй их в PNG и в итоговом PPTX: знак должен быть читаемым, не обрезанным и выровненным с соседним текстом.

## Рекомендуемый каркас

```html
<section class="slide">
  <div class="panel">
    <h1>Заголовок</h1>
    <p>Основной текст</p>
  </div>
  <img class="hero" src="hero.png" alt="" />
</section>
```

## Процесс

1. Реши, что PPTX нужен, до глубокой визуальной проработки.
2. Работай внутри отдельной директории презентации с локальным Node/npm.
3. Скопируй весь каталог `scripts/` навыка в `scripts/` этой презентации.
4. Создай `asset-manifest.json` и пройди `asset_gate_check.mjs`, если в презентации есть protected visuals. Для презентации без предметных визуалов используй пустой `visuals`.
5. Собери слайды как ограниченный HTML (`constrained HTML`), то есть HTML с заранее заданными размерами и простыми ограничениями.
6. Экспортируй все слайды в PNG и пройди `slide-qa.md` с реестром `exports/slide-qa.md`:

```bash
node scripts/export_deck_png.mjs --slides slides --out exports/png
```

7. После чистого прохода `1..N` запусти PPTX-экспорт:

```bash
node scripts/export_deck_pptx.mjs --slides slides --out exports/deck.pptx
```

7. Проверь PPTX:
   - текст редактируется;
   - первичные шрифты безопасны для Windows/macOS;
   - изображения на месте;
   - эмоджи, если они есть, читаются и не ломают интервалы;
   - `<meta charset="utf-8" />` был у каждого HTML-слайда;
   - объекты не выходят за слайд;
   - визуальные упрощения приемлемы.

Если после PPTX-проверки меняешь HTML, общие стили, изображения или размеры блоков, снова экспортируй PNG и обнови `exports/slide-qa.md` до чистого прохода.

## Существующий богатый HTML

Если исходная презентация использует сложные эффекты, которые работают только в браузере, не обещай идеальный редактируемый PPTX. Создай отдельную ограниченную PPTX-версию, то есть версию с понятными ограничениями для экспорта в PowerPoint. Если полезно, отдай рядом точные PDF/PNG, то есть файлы, которые лучше сохраняют внешний вид.

## Частые ошибки

- Текст напрямую внутри `div`.
- Контент выходит за границы `body`.
- Визуальный эффект висит на текстовом теге.
- `background-image` задан на `div` для важной картинки.
- PowerPoint подменяет шрифты иначе, чем браузер.
- Эмоджи стали слишком мелкими, обрезались или нарушили интервал строки после экспорта.
- У HTML-слайда нет `<meta charset="utf-8" />`.
- Слишком тесные текстовые блоки (`text boxes`).

Редактируемый экспорт (`editable export`) - это ограничение дизайна, а не последний шаг.
